敏捷流程

- IPM
- Kick-Off
- Developing
- DC
- Test on QA
- Test on UAT、PT
- Deploy Production
- After on Production

## Tech Lead工程师



|                          | 关键任务                                                     | 关键交付                                                     | 任务规则 |

| IPM                      | 1. 但开发对方案有疑问时，TL 需要从技术角度解释<br/>2. 宣讲迭代技术方案 （IPM 会议后） | 1、技术卡 AC 澄清                                            |          |
| Kick-Off                 |                                                              |                                                              |          |
| Developing               | 1. 负责对开发过程中的方案问题进行解释（救火）                |                                                              |          |
| DC                       |                                                              |                                                              |          |
| Test on QA               |                                                              |                                                              |          |
| Test on UAT、PT          | 1. 配合 UAT、PT 测试                                         | 1. 性能、安全问题卡片（周四、周五分配解决）                  |          |
| Deploy Production        |                                                              |                                                              |          |
| After on Production      |                                                              |                                                              |          |
| 预案（特殊场景）         | 1. 休假情况。准备和发展一个 backup，在休假前同步给他和团队。<br/><br/>2. 新人培训。准备 onboard 材料，让新人完善和更新它，找一个 buddy。<br/><br/>3. 安全事件。制定一个安全事件预案，第一时间和安全团队联系，并上报。 |                                                              |          |
| CR每天                   | 1. 制定 CR 清单<br/>2. 负责 CR 纪律，确保每日发生&nbsp;<br/>3. 督促 CR 中的问题整改完成<br/>4. 负责代码规范制定<br/>5. 负责规划组件和公用代码，为开发人员提供一套编程范例<br/>6. 提醒前端开发更新组件库 | 1. CR 清单<br/>2. 代码规范                                   |          |

| 日常，<br/>每个迭代 1 次 | 1. 组织技术分享                                              | 1. 技术话题的清单<br/>2. 会议日历预定                        |          |



## 研发工程师



|                     | 关键任务                                                     | 关键交付                                                     | 任务规则                                      |
| ------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ | --------------------------------------------- |
| IPM                 | 1. 提出有疑问的估点、业务<br/>2. 提出对工作量分配的风险      |                                                              |                                               |
| Kick-Off            | 1. 预约、组织开卡<br/>2. 根据 AC 和卡片任务对任务有所补充，在 comments&nbsp; 备注<br/>3. 挪卡到 In Dev | 1、Trello Comments<br/>2、挪卡                               |                                               |
| Developing          | 1. 根据 TL 准备的模型，进一步细化方案，设计 API 或者组件<br/>2. 任务拆分<br/>3. 编码<br/>4. 实现自动化测试<br/>5. 开发自测<br/>6. 挪卡到 Dev Done | 1. 业务代码<br/>2. 测试<br/>3. tasking 清单<br/>4.&nbsp; Sonar 扫描确认无问题 |                                               |
| DC                  | 1. 部署到 DEV&nbsp;<br/>2. 准备测试数据<br/>3. 预约或组织结卡<br/>4. 将需要上线变更的配置记录到上线卡<br/>5. 挪卡到待测试 | 1. Dev部署<br/>2. Dev配置更新                                |                                               |
| Test on QA          | 1.&nbsp; 配置变更                                            |                                                              |                                               |
| Test on UAT、PT     | 1. 上线 Owner 拉取分支，打标签<br/>2. 部署到 UAT、PT<br/>3. 数据迁移<br/>4. 配置变更 |                                                              |                                               |
| Deploy Production   | 1. 上线 owner 执行上线过程（按照上线流程执行）               | 1. 上线检查清单表格                                          |                                               |
| After on Production | 1. 上线 owner 执行 on call （周末、晚上）                    | 1. 生产问题的卡片<br/>2. drive 生产问题复盘报告<br/>3. 找相应的人解决 | oncall 期间外出需要带笔记本，或给其他同事代办 |
| 预案（特殊场景）    | 1. 休假。需要提前和 IM、TL 交接工作。                        |                                                              |                                               |



## 架构师

|                     | 关键任务                                                     | 关键交付                                                     | 任务规则 |
| ------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ | -------- |
| IPM 前              | 1. 对开发和 TL 的设计方案进行评审，守护架构。<br/><br/>2. 邀请业务参与技术评审 | 1、输出架构决策日志（代码库中使用 MD 日志记录）              |          |
| IPM                 |                                                              |                                                              |          |
| Kick-Off            |                                                              |                                                              |          |
| Developing          | 1. 负责对开发过程中的架构问题进行识别                        | 1. 输出架构相关技术卡                                        |          |
| DC                  |                                                              |                                                              |          |
| Test on QA          |                                                              |                                                              |          |
| Test on UAT、PT     | 1. 同步刷新架构图、部署图、模型图                            | 1. 按照版本刷新一次主要的架构文档、模型图 （主要的材料，通过复制一份操作，不修改之前的架构材料） |          |
| Deploy Production   |                                                              |                                                              |          |
| After on Production |                                                              |                                                              |          |
| 日常                | 维护架构文档                                                 | 1. 建立架构文档库，包括：<br/><br/>更新领域模型 (需要和产品团队共同完成业务梳理并产出业务架构)<br/>更新架构图 (含ER，UML，代码示例，数据库设计，API设计)<br/>更新部署图 (需要包含公有云，私有化部署的区别说明，水平扩容方案)<br/>输出技术架构更新的工作方式，更新要求，并在下下个迭代有序进行<br/>架构守护规范和工程实践要求，如何保证实际编码和架构设计是一致的<br/>在迭代结束时，给团队全员做一次分享 |          |



## OPS

|                                                              | 关键任务                                                     | 关键交付                                                     | 任务规则                                                     |
| ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| IPM                                                          | 1. 识别是否还存在DevOps支持的工作，如添加联调环境、添加域名解析等<br/>2. 识别是否涉及私有化部署相关的变更 | 1.新建DevOps卡片或者CheckList<br/>2. 更新私有化部署文档      |                                                              |
| Kick-Off,&nbsp; &nbsp;Developing,&nbsp; &nbsp;DC,&nbsp;&nbsp;Test on QA | 1. 解决流水线和环境中出现的问题                              | 1. 将常见问题维护至运维手册<br/>2. 总结常见问题，按需创建新的监控报警规则 |                                                              |
| Test on UAT、PT                                              | 1. 同步部署私有化环境（需要排卡0.5）<br/>2. 技术例会更新上线手册中的DevOps相关内容 | 1.可测试的私有化部署环境<br/>2. 更新私有化部署文档中需要执行的接口、脚本和改变的配置<br/>3. 上线追踪卡中的DevOps以及配置更新 |                                                              |
| Deploy Production                                            | 1.&nbsp;提供上线支持，维护上线流程，并记录上线过程中出现的问题 | 1. 更新后的上线流程记录和问题记录<br/>2. 将常见问题更新至运维手册 |                                                              |
| After on Production                                          |                                                              |                                                              |                                                              |
| 日常                                                         | 1. 解决流水线问题<br/>2. 解决DEV, QA, INT, UAT环境问题<br/>3. 分析生产出现的问题<br/>4. 维护阿里云资源 | 1. 将流程操作维护至运维手册<br/><br/>2. 维护整理常见问题的分析解决步骤<br/><br/>3. 每周五查看阿里云资源使用情况，并按需新购资源包和申请财务经费（每季度4W） | Infra as Code<br/>将配置和脚本放到repo中，可以快速重新构建对应的基础设施服务，减少手动执行和重复编码的情况。 |
