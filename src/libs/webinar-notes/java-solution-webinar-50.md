## 问题和背景

- 技术经理的岗位职责有哪些？
  - 技术：技术选型、技术问题、技术难题、技术创新
  - 业务：进度管理、和业务协同
  - 团队：冲突解决、人员绩效、团队培养
  - 参考模型：Expert、Trainer、Manager、Integrator
- 常见公司的技术经理的定位是什么？
  - 服务于公司的类型：服务型公司（项目型）、产品型公司（服务业务）。
  - 这两种公司形态对技术经理定位差异不大，当时作为技术经理来说体验不一样。例如管理方式是敏捷还是瀑布，对项目延期和验收是否敏感等。
  - 通常来说技术经理的定位：满足所服务的业务需求，承担团队技术决策和开发团队管理的职责，并解决技术和团队问题（俗称救火）。
- 有哪些人容易被任命为技术经理？
  - 认知转变：**不是一个兵，是一个小将的定位。**。比喻：**技术经理更像武官能打胜仗，带一点管理属性；单独的管理岗位更像文官，审时度势，协调资源。**
  - 技术能力扎实。能解决一些困难问题，能做技术方案，技术演进路线，将业务需求分解为能执行的任务。
  - 领导力比较强，能带领大家完成工作，协调工作任务。
  - 主动性。团队遇到问题时主动站出来解决。
  - 责任心。跟进一个问题能放到心上最终推动解决，不会撂挑子。
  - 目标导向。领导给出一个问题，聚焦问题目标，并及时给领导反馈。
  - 汇报和向上管理的能力。能清晰透明的反馈当前的工作状态。
  - 神秘的 "sense，懂事儿"：理解和揣摩干系人或者团队相关人员的意图、动机和利益关系，才能协调团队，建立领导力。能理解领导的意图和需求。
  - 谈判能力：和干系人掰扯，并最终让结果走向好的方向；能和外部团队进行沟通；向领导获取资源的能力。
  - 保护团队和争取团队利益：如果老板着急直接给开发提出额外的需求，或者外部团队对待团队成员不好如何保护团队。
  - 具备一些能背书的背景和资历：比如大厂、高学历、开源作者、外部影响力、写书。
  - 识人和任命的能力：如何理解团队的认知，建立团队的工作方式，合理的分配任务，让团队作为一个系统运转。
  - 管理能力：任务拆分、评估工作量、制定计划、划分优先级。
  - 机会和运气：直属领导升职；开辟出新的项目机会或者产品线；跳槽（比较坑）。
  - 能表现出上述能力，让领导相信你能做到上述相关方面；汇报上；写博客等。表达出相关能力和意愿。
- 那些人不会被任命为技术经理？
  - 只关心技术，不关心技术背后的问题。
  - 不关心产品的需求。例如当来了一个需求，需要考虑一堆东西。需求的合理性、成本、对架构的侵入、时间紧迫性、质量、优先级等众多需要权衡考虑的东西。
  - 不关心团队。只能自己一个人干。
  - 不关心项目管理，比如做计划之类的。
  - 不关心关心汇报，领导压根不知道你在忙什么。
  - 协作关系不好，和团队内外部人员之间的关系不好（尤其是神奇的东方文化）。
  - 非常 "Nice"，没有立场，无法保护团队，无法给团队带来利益，一定不要成为一个"好人"，即使面对老板（私底下可能会有一些协调）。
- 技术经理的日常工作有哪些？（如果你成为了一个新团队的技术经理，或者接管了一个团队，有可能是企业之外空降，该如何做？）
  - 理解当前的工作环境
    - 干系人管理和梳理，有哪些人和你相关的人，这些人的影响力和影响意愿如何？以及包括团队成员、角色。 
    - 梳理当前的工作流程，比如精益、敏捷等方法。
    - 当前的业务知识、架构状态、技术债
  - 确定目标，设定自己的 OKR，面临的困难是什么
  - 搭建适合自己（包括公司）的团队结构、工作方法和流程 （Way of working）。比如，需要多少个前后端的开发、QA，当前团队能接受多少需求，如何评审需求，如何做技术方案，如何评审技术方案，如何管理技术标准、约定等。
  - 技术方案、技术预研或者架构决策
  - 上下游团队沟通、外部系统联调沟通
  - 关键的代码编写
- 技术经理需要花多少时间写代码？
  - 技术经理一定要写代码，但是不能只关心写代码。
  - 30% - 50%
  - 甲方单位写代码更少，乙方更多。
- 如何向上管理？如果领导给出一个无法完成的任务怎么办？
  - 实例：当前政策原因，需要紧急在一个月内完成某个政策合规的需求，必须在特定时间内完成，但是，团队的容量不足，或者又遇到了流感请假，怎么办？
  - 方法：
    - 砍低优先级需求，业务方可能觉得都应该做。
    - 申请人力，流程不可能这么快走完。
    - 申请加班，团队对加班意愿不强。
    - 降低质量，曲线救国，能手动配置的就不要做个功能。
    - 请求支援，分给其他团队接一些。
    - 再次分析业务，通过分析业务使用低成本解决问题。
  - 分析：**关键要分析矛盾，找到主要矛盾，以及关键领导在乎的矛盾。** 想要得到什么，就想要看看有什么可以丢掉或者取舍，或者牺牲。
- 如何向下管理？如果团队成员不听指挥怎么办？
  - 举一个例子，在代码评审时，有一个同事就是喜欢写代码压行（或者使用项目中从来没有遇到的新特性），不按照团队的代码规范执行或者其他同事的反对意见，并声称他的方案更好。这种情况如何处理？
  - 方法：
    - 使用一些优缺点进行理论，达成**共识（共识并非对错）**。可能会得到一些模棱两可的结论（可能限于时间），无法取得共识。
    - 技术经理独裁，由技术经理决策完成，并强制执行。副作用是会打击团队成员提出自己想法的积极性，决策的成功依赖于技术经理的经验以及对问题的理解。
    - 使用团队其他人影响他，避免技术经理下场去和团队成员 battle，技术应该引导整个团队使用团队的立场和态度推动共识。
      - 技巧：
        - 如果违反一个代码风格，确实大家都觉得很好，提出修改所有地方的"成本"有多大，如果都能改完，就改。
        - 建立一个团队决策的机制，这个机制不依赖某个人完成技术决策（技术经理有影响的空间或者更大的权重，但不要把决策权直接和技术经理自己钩挂）。例如，建立一个技术决策的会议，记录需要决策的点，技术经理通过投票引导获得共识。关键点：**所有人参与到决策过程中**，并且决策需要做到**流程正义**，避免决策失败，后面翻旧账。
  - 补充
    - 就是有一些人不断挑战团队，不听指挥怎么处理？举个例子，就是不满意打卡制度，就是不打卡。
      - 在什么条件下合情合理的行使管理职能（最差的情况，明确证据或者事实表明问题的时候，而是觉得可以），进行谈话→绩效反馈→移出项目组或者开除。
      - 根据这些问题制定和更新补充规则、制度、规范，当再次触发这些规则时，可以**合情合理**行使管理职能。
      - 招人很容易，开人其实很难。
    - 技术经理提高自己的技术影响力，树立技术专家的形象。
    - 技术决策需要整理和记录。
- 如何平行管理？如果公司其它部门同事不配合怎么办？例子，如何同产品、业务、测试扯皮？
  - 例子1，业务提出了一个需求，产品要求按照一定方式实现，但是技术上，成本很高，实际上有没有什么用户真的会用，这时你改怎么办？
    - 处理方式和沟通技巧
      - 对人：
        - 基于组织架构，分析干系人，了解直属老板和大老板的关系，找到有共同利益的人进行合作。在这个例子中，产品一定要做某件事，可能背后有其他原因，比如可能有一些 KPI 方面的诉求。
        - 社交关系的影响，人情社会，**例如咖啡、小礼物的。**
        - 构建差异感，重新定义"我们"的词的范围。
      - 对事：从公司产品投资角度分析是否对市场和用户具有更大价值；从研发流程上进行管理根据研发节奏、排期、估算的管理，给出成本，让产品经理评估其价值。
      - 本质问题：
        - **技术经理需要知道自己的影响力、权力构成。** 技术经理无权拒绝需求，但是可以给出成本。
        - 分析对方的**动机**。
  - 例子 2：其它部分需要协同上线，但是他们赶不上或者不愿意配合，怎么办？
    - 常规情况和例子 1 类似，需要使用强制行为（升级），使用"威逼"的武器。使用责任驱动，当己方上线后，如果不配合的团队出现的任何风险不予承认，并通过邮件通知，抄送领导。
    - 通过暗示展现展现可以伤害对方的能力和意愿，暗示对方可能承担责任和不良后果。
    - 注意：**无论如何不应出现正面冲突。** 通过借力的方式和对方非直接冲突，举个例子，把其它相关干系人拿出来去和其沟通，比如产品经理层面、业务层面，最后甚至领导层面。在团队中没有权力去影响 peers。
    - 特殊情况：被领导当做作为黑脸用，语言上面更委婉。黑脸并不是坏事，可能看做一种信任，可能存在利益补偿。
- 作为甲方公司，如何管理供应商（管外包）？
  - 人天型：能力比较差，没有主动性，感觉比较**听话**，胆小不思考。认知分级：混乱（认知高）、复杂（经验多）、困难（智力强）、简单（普通，都可以干）。只购买人员工作时间，不对交付物负责，不具有 ownership，产物需要评审，通过高低搭配评审质量。
    - 思考：为什么一家公司需要购买人天服务？一线城市：800 - 6000 之间。
      - 项目波动和财务预测：劳务市场可以更为灵活的安排人员，某些行业人员成本在计税上没有优势。
      - 企业规模限制：企业不批准足够的人员数量。
      - 人事裁员成本：外企、国企或者其他领域裁员对公司口碑影响很大。
    - 如果外包人员不满足需求，或者不听指挥怎么办？**提出换人即可。**
  - 项目型：外采的项目如何管理，如何减少风险、按时交付、达到交付质量、满足技术规范？负责背锅就行了，没啥可以管的。
    - 编写和确定需求规格，定标参数的时候。 
    - 对核心人员需要**定级面试**。
    - 联合开发，管理权交给乙方项目经理，插眼。
    - 只需要看结果，不用关心工作方式（乙方团队制定） ，通过甲方技术委员会评审。
    - **周期性 check in（一般按周、按迭代）**
    - 代码检视。
    - **定标参数很多时候不会非常具体，过程中提出一些要求，需要一些谈判、管理和关系。** 尽可能手上有一些管理手段，比如评审委员会等，或者一些利益交换。
  - 逻辑：两种外包方式的合同不一样，人力型约束的是**人员工作时间**（通过面试），项目型约束的是**交付物**。
- 作为乙方公司，如何影响甲方客户（管爸爸）？
  - 对甲方进行分类讨论：
    - 知道自己想要什么的，很多决策甲方能决定，不容易影响。
    - 不知道自己想要什么的，容易被乙方影响。
  - 甲乙双方始终都在博弈，搞清楚甲方的干系人、利益关注点、软肋（他们 KPI 中关心什么，担心什么，为什么而焦虑），尤其是找到甲方的一些制约（比如业务方的一些要求、上一级领导的要求）。
  - 如果甲方提出一些无理要求，例如增加合同约定内容的任务？人性的弱点：甲方的人往往并不需要为甲方公司获取的利益，多半是为了自己少一些事儿。
    - 如果甲方同意对需求进行一些置换，可以进行置换。
    - 如果甲方同意投入开发资源，比如开发人员，可以代为管理。（可以代管，但是人要出，甲方经理有时候无所谓）
    - 计算增加的内容是否会影响交付进度，如果交付进度对甲方领导有考核要求，可以优先做这一些东西，再谈置换。
    - 如果增加的内容影响比较大，可以给上一级领导反馈，暴露项目风险。
  - 如果甲方在结项时提出一些整改项（修复一些安全漏洞），而这些都是两方没有考虑到的？
  - 如果甲方对交付物质量不满意，实际上质量可能会很高，怎么办？
  - 如果甲方自己的需求给晚了，但是要求团队加班按时交付怎么办？
  - 如果甲方给出的需求质量太低，不符合逻辑，或者有很多错误，怎么办？
- 如何面对能力危机，如果团队成员能力比你更强怎么办？
  - 很有能力，又比较服从安排：皆大欢喜，如果有好的机会，可以推荐更好的岗位。
  - 很有能力，但是和团队对抗，不听取意见（刺头）：例如，来了一个华为、阿里、大厂的人，对小团队的做事方法或者技术质疑。
    - **识别出发点**，如果出发点是好的，采纳或者推广，如果出发点不好，采用一些管理技巧。参考"态度-能力模型"。![img.png](./java-solution-webinar-50/img.png)
    - 如果出发点好，采用谈话：所有好的东西，都有成本，这些成本不合适所有公司。一些基础设施搭建、分支管理方法等。
    - 如果出发点不好，采用一些方法，例如权责一致，如果他坚持自己的意见更多，给予更多的责任，用责任来管理，使之和表现能力匹配。 谁说的让谁做，避免光说不做。
  - 自以为很有能力，但是其实经验比较少。【谈话或者直接进入绩效关注】
  - 自以为很有能力，有很多意见，但是不被团队、领导认可，也不会挑战领导和团队的做法。做一些谈话，或者引导，对团队伤害不大。
  - 平时不被发现，实际上在团队中非常有能力。一般来说自信心不多，可以作为团队的核心贡献者，非常可靠的一类人。
- 如何面对团队增长？团队人数过多怎么办？
- 如何面对客户的批评和投诉？
**- 作为甲方公司技术经理，如何管理供应商？**
- 如何面对不确定性，团队成员变动怎么办？一个核心人员离职，他之前负责了一些很重要的模块，很多逻辑只有他知道。
  - 建立设计+评审的机制，需要参与所有的方案评审过程，以及文档要求。
  - 代码公有制。每日 Review 代码，所有人互相知道相互的代码是什么。
  - 不要搞代码私有田。不要一个人固定做一个模块，造成代码可读性、设计性非常差，其他人没法接，"代码没有曝光在别人的眼睛下，小动作比较多，有一些非常糟糕的实现"。【权衡一下效率】
  - 关键岗位设置 A/B 角，例如技术经理请假，以后升职，会有一个 backup 人员存在。
- 如何处理团队成员矛盾，如果团队成员之间存在矛盾、感情问题怎么办？例如两人在团队上谈恋爱，然后又失恋。
  - 个人情感问题：避免安排情侣在关键的岗位，例如前后端 TL，这种场景。
  - 个人情绪问题：两个人在项目上总是吵架，有私人恩怨的。
    - 收集确定的事实、证据，确定这件事存在，反馈给 HR，人员关注或者调动。
    - 越好的公司这种情况越好。
- 被投诉了怎么办？如何沟通？例如，被团队成员给领导投诉，说过于严厉、任务分配不公平等。
  - TODO，暂时没有想到合适的应对方法
- 如何处理进度压力，当遇到延期怎么办？如何让团队成员加班？如何驱动团队？例如，如果团队成员承诺了 3 天完成，就是 5 天才完成，并无所谓，怎么办？
  - 安排加班，时间可以调休，一部分人就会接受。
  - 隐形推进度，推动团队主动加班，每天把每个人的任务按照人天倒推完成时间，并在晨会"曝光"。
  - "曝光"是一个非常重要的技巧：1. 公开夸奖做的快的同事。2. 刻意多检查任务没有完成的同事。
  - 加班的同事增加一些额外的奖励（物质或者精神方面的）
  - 从其它方面将交付压力传递给团队，技术经理不要将交付风险和压力屏蔽掉。
  - 警惕：**如果团队一直都做不完，会导致大家觉得做不完也无所谓，所以在迭代规划的时候，需要尽可能让每个迭代都完成。** 如果加班成为常态，大部分人都在划水，再也无法通过驱动加班完成紧急任务，因为**做不完也没有什么后果。** 
  - 非常糟糕的场景：**公司故意分配额外的工作给团队成员，团队成员做不完也没有责任，进入恶性循环。**
- 当直属（+1 的领导）领导升职后，不再作为直属领导（+2 的领导，变成领导的领导，但有时候会给你派活儿），这种关系如何处理？
  -
